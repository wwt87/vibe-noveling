# 剧情规划

## 你要产出什么

`chapters/vol-{v}/ch-{c}/故事梗概.md` + `大纲.md`（经 HTML 可视化审阅后写入）：

- **故事梗概**：3-8 段因果链叙述，对话中展示并调整后暂存
- **剧情思路卡**：为什么这样排。含核心兑现、主角状态变化、核心冲突链、信息释放策略、本章边界
- **可写场景纲要**：3-6 个叙述式场景块，每块 1-2 段散文，场景目标/触发事件/冲突落地/转折点/落点变化/细节锚点嵌入文中
- **HTML 可视化审阅**：故事梗概确认后生成 `synopsis-view.html`（因果链流程图），大纲确认后生成 `outline-view.html`（纵向场景卡片），用户分别在浏览器中编辑后复制 Markdown 回对话，再写入最终 `.md` 文件

**不要产出**：场景正文内的字段标签、剧情点编号、情绪走向栏位、场景类型栏位、字数配额、面向 agent 的执行表。

## 开始前：回答三个根问题

在输出任何规划内容前，先在 thinking 里回答。**答不上就停**——上下文不够：

1. **这一章为什么必须存在？** 跳过它读者损失什么？
2. **本章的本质变化是什么？** 从起始到结束，主角/局面发生了什么样的具体变化？
3. **主角的牛逼体现在哪？** 本章给了他什么主动推动局面的机会？他的主观能动性怎么体现？

这三个答案是后续所有输出的根。如果后续和它们矛盾，回头修正。

## 前置：目录约定与风格基线

1. 章节目录统一为 `chapters/vol-{v}/ch-{c}/`，本章所有产物写入同一目录
2. 读取 `CLAUDE.md` 的默认创作风格基线：主角定位、装逼主驱动、情节取舍规则、默认爽点循环。含"小说宪法"时宪法优先
3. 确定章节编号与所属卷：从 CLAUDE.md 创作进度读取或用户指定，无法定位 volume_padded 时先提示补卷信息

## 上下文读取（固定边界）

1. 当前事件 → `memory/future/40-events/` 对应文件，读取事件故事大纲和事件高潮位置，用于确定本章在事件弧线中的职责
2. novel-knowledge 搜索事件中出现的角色/地点/物品，读命中 entity 文件
3. `memory/future/20-threads.md` + `90-sync-tracker.md`
4. 上一章 `chapters/vol-{pv}/ch-{pc}/正文.md`
5. `memory/past.md` → **停，不继续展开**

修订模式额外先读取已有 `大纲.md`。

20-threads 和 90-sync-tracker 弥补 past.md 伏笔摘要的颗粒度不足：past.md 给概要，追踪表给具体来源章节和计划回收位置。两者结合后规划时不再回读更早章节正文追伏笔。读到 past.md 后不要因其中设定名词继续扩读。

## 模式判定

- 从零设计下一章 / 当前章没有 outline → **新规划模式**
- 已有大纲上调整 / 补桥接 / 改结尾 / 重写后半章 → **修订模式**
- 修订模式下默认不回到从零发散；只有骨架失效或用户明确要大改时才退回

## 新规划模式

### 1. 定节奏

先定位本章在全书结构中的角色。过桥章就直说过桥——把角色从 A 送到 B，保留钩子，说明在为下一次装逼兑现做什么蓄势。

```
📍 所属卷：Vol-02 | 主母型：Institutionalized | 卷内位置：中前段
本章负责：1. ... 2. ... 3. ...
本章不负责：1. ... 2. ...
密度定位：高/中/低 ← 根据事件高潮位置和本章职责判定（高潮段=高，过桥=低，常规推进=中）
```

卷级节奏信息不足时，提示先回 novel-bookplan 补卷卡，而非在信息不全时强推单章。

### 2. 定任务

生成章节任务卡，输出前自问：
- 是否服务主角装逼链条？
- 主角装逼兑现是什么？
- 谁承担压制/打脸作用？
- 什么跟主角装逼无关可以延后？

```
📌 本章主任务：...
必须推进的线：1. ... 2. ...
允许延期的线：1. ... 2. ...
建议收尾点：...
```

### 3. 定起落

- 起始状态从上一章正文真实提取，不可编造
- 结束目标从当前事件职责、线程状态、未回收伏笔推导
- 写清本章负责什么、不负责什么

### 4. 故事梗概

用 3-8 个自然段落叙述本章从起到落的因果链。**必须是因果链，不是事件列表**——"因为 A，所以 B；做 B 时撞上 C；因此局面变成 D"。说清关键转折和信息释放节奏，暴露结构问题。

不编号，不填字段。写完后自问：**我是在讲故事，还是在列条目？** 如果读起来像说明书或执行清单，重写。

**流程**：先在对话中展示故事梗概草稿 → 用户提出调整 → 修改后再展示 → 用户无异议后暂存内容（不写文件），待步骤 9 生成 HTML 可视化。

### 5. 逐步澄清

不是系统性填表。哪里不清楚就问哪里：转场桥接、角色动机、信息释放时机、冲突落地方式。一次只问一个，每个带 2-3 个修法选项。能按既有约束直接局部修补的不要为凑数提问。

### 6. 设定完整度 + 命名一致性检查

检查涉及实体的设定文件是否齐全。缺失的写入 `缺失设定.md` 并提示补全。

检查本章称呼是否能映射到已有实体 canonical name。

### 7. 生成双层大纲

#### 剧情思路卡（固定 5 项）

- `本章核心兑现`：不是摘要，是作者意图
- `主角状态变化`
- `核心冲突链`：写因果链，不写结论。例："因为苏寒染要尽快摸清夜市门道，他接受周华顺带路；周华顺借带路试探他是否懂行；因此双方从单纯交易转成互相试底"
- `信息释放策略`：明给什么 / 压住什么
- `本章边界`：这章不负责什么

#### 可写场景纲要（3-6 块，叙述式段落）

标题语法：`### 场景 {N}：{场景名} | 密度：{高/中/低}`。正文为 1-2 个自然段落，将场景目标、触发事件、冲突落地、转折点、落点变化、细节锚点全部嵌入散文中。段落内禁止字段标签（"场景目标："等）。

写法铁律：
- 触发事件写清**谁做了什么**场面才启动，不写"随后""经过一番周折"
- 冲突落地写成动作/试探/打断/压价/验货/插手等可见行为，不写"双方开始博弈"
- 落点变化落到关系/判断/风险/目标/信息占有中至少一项，不写"关系发生变化"
- 细节锚点每场保留 1-3 个硬信息，不允许只写空泛氛围

**禁止出现：**
- "展现出""体现了""暗示了""为……埋下伏笔"这类叙事功能评论
- "双方开始博弈""气氛愈发紧张""关系发生变化"这类空泛结果句
- 大段心理分析、剧情点编号、面向正文执行的碎片化字段

### 7.5. 大纲 AI 味检测

对生成的叙述式 `可写场景纲要` 执行大纲级 AI 味检测。检测项详见 `references/planning-checks.md` 第 4 节。

检测方式：逐场景按 7 项顺序检查。单个场景块触发 3 项及以上 → 重写该场景块；触发 1-2 项 → 局部修复。全部场景块通过后方可进入展示/确认。

### 8. 展示 → 确认

先展示大纲草稿。确认后单独处理标题（2-3 个候选，允许"待定"）。标题确认后暂存内容，不立即写入文件。

### 9. HTML 可视化与用户审阅

#### 步骤 9a：故事梗概 HTML

故事梗概确认后，读取 `{SKILL_DIR}/templates/synopsis-viewer.html`，将梗概内容注入 `PLAN_DATA` JSON，写入 `chapters/vol-{v}/ch-{c}/synopsis-view.html`，用 `open` 打开。

告知用户："故事梗概已在浏览器中打开。审阅编辑后，点击「复制故事梗概」按钮，粘贴回此对话。"

收到用户粘贴的梗概后，写入 `故事梗概.md`。

#### 步骤 9b：大纲 HTML

大纲和标题确认后，读取 `{SKILL_DIR}/templates/outline-viewer.html`，将大纲内容注入 `PLAN_DATA` JSON，写入 `chapters/vol-{v}/ch-{c}/outline-view.html`，用 `open` 打开。

告知用户："大纲已在浏览器中打开。审阅编辑后，点击「复制大纲」按钮，粘贴回此对话。"

收到用户粘贴的大纲后，写入 `大纲.md`，继续正文试写验证。

## 修订模式

1. 读已有大纲，输出修订简报：当前骨架、建议保留、待解决问题
2. 给 2-3 个修订策略，推荐最稳的
3. 确认变更边界：保留什么 / 删除弱化什么 / 新增什么 / 收尾改成什么
4. 只重做受影响段落，未受影响的默认沿用
5. 局部修改破坏整章推进链时才退回新规划模式
6. 修订后做联动检查：前章承接、本章职责、伏笔状态、下一章起手

## 正文试写验证

大纲确认后，主 session 全文试写正文（2500-4000 字，覆盖全部场景块）。

试写稿保存到 `Opus试写.md`，反推报告保存到 `Opus报告.md`。报告必须回答：
- 哪段最难自然写开？为什么？
- 哪个转场需要补桥接？
- 哪个角色动机/语言依据不足？
- 哪里被迫脑补了？应回填大纲还是设定？
- 最终判定：可直接写 / 需小修后写 / 需先重做大纲

拿到报告后，把可行动问题整理成待确认队列，一次只问一个，每个带 2-3 个修法选项。当前问题确认前不进入下一个。如果判定需先回到修订模式，就在那一题后分流。待确认问题清空前不放行 novel-write。

## 禁止

- AskUserQuestion 工具
- 未经讨论直接改文件 / 未经确认保存大纲或补标题
- 一次性抛多个问题 / 只问问题不给方案
- 用低优先级文件覆盖高优先级信息
- 在大纲中写角色台词或语言风格细节（novel-write 的职责）
- 修订局部时顺手重写整章
- 用户说"在原大纲上改"时强行走从零规划
